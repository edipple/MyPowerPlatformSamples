1.1 Deployment and Configuration

Welcome to this code-component (PCF) Deployment and Configuration lab - you'll be creating a field code component that can be deployed to both canvas and model-driven apps.

Install pre-requisites

For this tutorial you need to have completed the previous lab 'Code components (PCF) Quick Start'.

Understanding Build Modes

Ideally, you should only publish production builds of code components into Microsoft Dataverse. For larger code components, publishing a development build may result in the error Web resource size is too large. Since we are going to redirect the code component's bundle.js to a locally built version using Fiddler (see below), you can update your .pcfproj file to always build in production mode when using pac pcf push by setting the property PcfBuildMode to production.

  1. Open the DynamicTextInput project in VSCode.
  2. Locate the file DynamicTextInput.pcfproj and add the PcfBuildMode element to the DynamicTextInput PropertyGroup. This will ensure than when we deploy the code component to Microsoft Dataverse, it will be build using the production mode. The PropertyGroup should then look similar to:
xxxxxxxxxx
<PropertyGroup>
  <Name>DynamicTextInput</Name>
  <ProjectGuid>...</ProjectGuid>
  <OutputPath>$(MSBuildThisFileDirectory)out\controls</OutputPath>
  <PcfBuildMode>production</PcfBuildMode>
 </PropertyGroup>

  1. You can also build a version of your bundle.js manually to see the difference. Run the following at the terminal to create a production build. Note: If you still have npm start watch running, you'll need to stop it using Ctrl+C at the terminal it is running from.

    xxxxxxxxxx
    npm run build -- --buildMode production

    Locate the output at out\controls\DynamicTextInput\bundle.js and look at the code. Notice it is minified and does not contain any source maps that enable debugging in the browser F12 developer tools.

  2. Now run the following at the command line:

    xxxxxxxxxx
    npm run build

    Look again at the bundle.js and notice how the file is much larger with source maps. This version of the code should not be deployed but used purely for development/debug purposes.

Deploying to Microsoft Dataverse during testing and development

In the previous lab we used the test harness to debug the code component. This has several limitations and code components should always be deployed to Microsoft Dataverse to fully test. In some cases (e.g. when using advanced features such as the WebApi), it is the only way to test your component.

Create a solution and publisher

  1. Inside your Dataverse environment, ensure there's a publisher created with a prefix of samples. Open make.powerapps.com and select your environment.
  2. Select Solutions.
  3. Create a new solution using New Solution.
  4. Provide a display name & name (e.g. SamplesPCF)
  5. Select New publisher.
  6. Enter Samples as the Display name and Name.
  7. Enter samples as the Prefix and then select Save.

Equally, this could be your own publisher name, provided you update the publisher prefix parameter in the call to pac pcf push below. More information: Create a solution publisher.

  1. Select the newly created publisher.
  2. Select Create to create the new solution.
  3. Open the new Solution.
  4. Add Existing -> Table -> Account -> Next -> Add.
  5. You will now have a solution that contains only the Account table. Later we will use this to add our code component to the Account form.

Deploy your code component

Once you've saved the publisher, you are ready to authorize the Microsoft Power Platform CLI against your environment so that you can push the compiled code component.

  1. At the VSCode terminal, use:

    xxxxxxxxxx
    pac auth create --url https://myorg.crm.dynamics.com

    Replace myorg.crm.dynamics.com with the URL of your Dataverse environment. You can get this URL using the Developer resources link under the gear settings in make.powerapps.com.

Note: Ensure you sign in with a system administrator when prompted. The privileges provided by these roles are needed to deploy any code components to Dataverse.

  1. To deploy your code component, use the following at the VS Code terminal:

    xxxxxxxxxx
        pac pcf push --publisher-prefix samples

    Note: If you receive the error Missing required tool: MSBuild.exe/dotnet.exe, add MSBuild.exe/dotnet.exe in Path environment variable or use Developer Command Prompt for Visual Studio Code. You must install either dotnet SDK, Visual Studio 2019 for Windows & Mac or Build Tools for Visual Studio 2019. Make sure to select the .NET build tools workload as described in the prerequisites. If you have a VSCode terminal open, you will need to close this before the dotnet sdk will be available.

  2. Once completed, this process creates a temporary solution named PowerAppTools_samples in your environment. The DynamicTextInput code component will be added to this solution. You can move the code component into your solution later if necessary. More information: Code Component Application Lifecycle Management (ALM).

Note: When you create code-component you provide a namespace that is added to the control manifest file. This is used when building the code and will be included in the bundle.js generated output. The publisher prefix on the other hand is not referenced in the code but is used to control which solution publisher owns a code-component and cannot be changed after deployment. Makes sure you set the solution prefix to the one that you want to use to deploy your code component right from the start.

Configure your code component on a model-driven form

  1. Open make.powerapps.com and select your Development environment.
  2. Open your Samples solution. This is the samples solution you created above, not the one automatically created by pac pcf push that is called PowerAppTools_samples.
  3. Select the Account table -> Form -> Add form -> Main form.
  4. Name the form PCF Form.
  5. Save the form.
  6. Switch to Classic (Note in the future this will not be necessary!).
  7. Locate the Description field in the Details tab.
  8. Select the field and use Change Properties.
  9. Select the Controls tab -> Add Control.
  10. Select Dynamic Text Input -> Add.
  11. Leave the properties blank other than then Text property that will be automatically bound to the description field. You could choose to change the font and font size here, otherwise the control will use the model-driven App default font style.
  12. Select the Dynamic Text Input as the default for Web, Phone, and Tablet.
  13. Select OK -> Save -> Publish.
  14. Navigate back to the solution, and select Apps -> Add -> New -> App -> Model-driven app.
  15. Select the Modern app designer (preview) (again, this will not be necessary in the future).
  16. Select Create and Name your app.
  17. Select Add page -> Table based view and form -> Next -> Select Account -> Add.
  18. Save -> Publish -> Play.
  19. In the model-driven app, open an account record (create one if you have none).
  20. In the dropdown below the account name, select your new form:
  21. Select the Details tab.
  22. You should now see your Dynamic Text Field code component, and be able to enter text with the height automatically resizing.
  23. Try moving the field to be above another field to see the automatic reflow of the form.

Configuring Fiddler for debugging inside Microsoft Dataverse

Once you have deployed your component to Microsoft Dataverse, normally with each change it would need to be re-deployed and the app reloaded. This can be time consuming and will slow down the development process, so instead we use Fiddler's AutoResponder feature to redirect the request for the code component to a local development build of your code component without having to continuously deploy changes as you debug your code. This also allows you to debug against a non-development downstream environment without having to first deploy a development build.

  1. Open Fiddler (which you should have installed as part of the pre-requisites). Note: If you get a message about 'AppContainer Configuration', select Cancel.

  2. From the menu bar, go to Tools, and then select Options.

  3. Select the HTTPS tab in the dialog box and check the Capture HTTPS CONNECTS and Decrypt HTTPS traffic checkboxes so that the HTTPS traffic is captured and then decrypted:

    When prompted, select Yes to Trust the Fiddler Root certificate?, Yes to Do you want to install this certificate? and Yes to Add certificate to the Machine Root List?.

  4. Select OK.

  5. In the right-hand panel, select the AutoResponder tab.

  6. Ensure that Enable Rules and Unmatched requests passthrough are checked.

  7. Select Add Rule and enter first:

xxxxxxxxxx
REGEX:(.*?)((?'folder'css|html)(%252f|\/))?SampleNamespace\.DynamicTextInput[\.\/](?'fname'[^?]*\.*)(.*?)$
  1. Enter the path to respond to:
xxxxxxxxxx
C:\[YOUR DEV PATH]\DynamicTextInput\out\controls\DynamicTextInput\${folder}\${fname}

Important: Where [YOUR DEV PATH] is the location of your own out\controls\DynamicTextInput folder. Don't forget to add the ${folder}\${fname} onto the end. E.g.: C:\src\DynamicTextInput\out\controls\DynamicTextInput\${folder}\${fname}

The regular expression is specially created to match both JavaScript and CSS resources in the locations both in canvas apps and model-driven apps.

It will look similar to:

More information Debugging Code Components.

  1. Select Save.
  2. Open the Filters tab and check Use Filters. In the Response Headers section, check Set response header and provide the following:
xxxxxxxxxx
Access-Control-Allow-Origin = *

Note: This step is only necessary when debugging code components after they are deployed to canvas apps because the resources are stored in blob storage rather than underneath the powerapp.com domain. As such, any requests to these resources will require cross-domain access when loaded by the browser. Only enable this Access-Control-Allow-Origin filter rule when you are debugging since it will modify headers of other sites that you visit.

  1. Open your project in VSCode and at the terminal start the build in watch mode:

    xxxxxxxxxx
    npm start watch

    This will ensure that the code bundle.js created locally will be a development build with source maps to enable debugging inside the browser.

  2. Now that you have the AutoResponder rules running you will need to first clear the cache in the browser and reload the page containing the code component. This can easily be done by opening developer tools (Ctrl + Shift + I), right-clicking the Refresh > Empty cache and hard refresh.

  3. You can now open the index.ts using Open file from the ellipsis’s menu (Ctrl+P). You can add a breakpoint in a similar way to when using the test harness.

  4. If you go back to VSCode and edit the index.ts, save, and refresh your browser, you will see the new file being reflected inside the browser without the need to re-publish. This is a very efficient way of developing code components.

    Note: If you need to update the manifest in any way, you will need to re-deploy a new version of the code component using the technique below.

  5. This configuration of Fiddler only needs to be done once, and remember that this mechanism will only work when you have Fiddler running.

Versioning strategies

Now that you've got the code component working inside Power Apps, you will eventually need to push a new version to Microsoft Dataverse so that you don't need to use Fiddler, and to ensure the production version of the code is used for other users.

Before you can use pac pcf push, you will need to ensure that the component manifest version is incremented so that Power Apps knows that there is a new version. This can be done in two ways:

Manually incrementing the version

  1. Close Fiddler if you still have it running.

  2. Locate the DynamicTextInput\ControlManifest.Input.xml file in VSCode.

  3. Under the control element, increment the minor version of the version attribute. E.g. 0.0.1 -> 0.0.2

  4. Add a new property that will be included in the new version:

    xxxxxxxxxx
    <property name="MaxHeightValue" display-name-key="MaxHeightValue" description-key="MaxHeightValue_Desc" of-type="Whole.None" usage="input"/>

  5. Save the manifest.

  6. At the terminal use push as before:

    xxxxxxxxxx
    pac pcf push -pp samples

    Note: This is using the short form of --publisher-prefix.

  7. Refresh your model-driven app and a new version will be automatically loaded, however to provide any new configuration parameters you will need to edit the form that contains the code-component and re-publish.

  8. To use the new version in your canvas app you will need to re-open your app. You will be prompted to upgrade your code component. Edit the app we created previously and you will see the prompt to upgrade the component. After selecting Update, the new version will be loaded into the app. Without performing this step, the app will continue to use the older version.

More information: Code components application lifecycle management - Power Apps | Microsoft Docs

Automatically incrementing the version

You can perform the increment of the manifest automatically using the following command at the terminal in VSCode.

xxxxxxxxxx
pac pcf version --strategy manifest

You will then need to push the new version as above. This is useful for automated build scripts that are running inside build pipelines, but is out of scope of this lab.

Try it out now, by running the above command and observing the version attribute incrementing. There are other strategies available that you learn more about: pac pcf version

Creating a deployment solution project

Once you have finished your code component and are ready to deploy, you can create a solution one of two ways:

  1. Add the code component that was deployed using pac pcf push to your own solution (provided it uses the same publisher prefix) and export that solution from Microsoft Dataverse.
  2. Build a solution directly from the code using pac solution init. This technique will create a managed or unmanaged solution without the need for deploying the code component to Microsoft Dataverse. To use this option, follow the steps described in Import components into model-driven apps in Microsoft Dataverse. This technique is out of scope of this lab.
